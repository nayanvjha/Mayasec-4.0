# SOC Console Quick Model Reference

## The Core Shift

**FROM:** Metric dashboard (charts, cards, summaries)  
**TO:** Event-driven investigation console (stream, details, context)

---

## Event Model (What the analyst sees)

```
Event {
  timestamp          // When it happened
  source_ip          // Where it came from
  destination_ip     // Where it went to
  event_type         // What kind of threat
  threat_score       // 0-100 urgency
  severity_level     // CRITICAL/HIGH/MEDIUM/LOW
  raw_data           // The actual data from backend
}
```

**Analyst reads:** Newest events first, scans source IPs and severity, picks one to investigate.

---

## Alert Model (When events become actionable)

```
Alert {
  event_ids          // Array of related event IDs
  primary_event_id   // The event that triggered this alert
  status             // OPEN, INVESTIGATING, RESOLVED, FALSE_POSITIVE
  created_at         // When the alert was generated
}
```

**Relationship:** Multiple events → One alert (grouped for investigation)

**Example:**
- Event 1: Port scan from 10.1.1.100 (12:00:00)
- Event 2: Port scan from 10.1.1.100 (12:00:15)
- Event 3: Port scan from 10.1.1.100 (12:00:30)
- Event 4: High-severity port scan from 10.1.1.100 (12:00:45) → **ALERT CREATED**

All 4 events belong to the same alert.

---

## Analyst Flow (The questions they ask)

1. **"What's happening?"**
   → Look at Live Event Stream (newest first)

2. **"Is this real?"**
   → Click event → See details, threat intel, raw data

3. **"Is this part of an attack?"**
   → Click "Show Timeline" → See all events from same source IP

4. **"What's the context?"**
   → Click "Show History" → See previous alerts from this source

5. **"What do I do?"**
   → Mark alert: True Positive / False Positive / Investigating

---

## Console Layout (Information Architecture)

**Layer 1 - Live Stream (Always Visible)**
- Newest events first
- Tap to select

**Layer 2 - Event Details (On Select)**
- Full event data
- Threat intelligence
- Options: Show Timeline, Show History, Investigate

**Layer 3 - Timeline (On Demand)**
- Related events from same source IP
- Time order (oldest to newest)
- Shows attack evolution

**Layer 4 - Historical Context (On Demand)**
- Previous alerts from this source
- Threat intel reputation
- Organization context

---

## What Updates Live (WebSocket)

✓ New events (instant display)  
✓ Event severity/score (if recalculated)  
✓ Connection status  
✓ Analyst presence  

---

## What Does NOT Update Live

✗ Event counts (derived from stream)  
✗ Statistics (calculated, not polled)  
✗ Summary aggregations (frontend calculates)  
✗ Metric cards (deprecated)  
✗ Dashboard refreshes (polling is dead)  

---

## Data Authority

**Backend is Source of Truth:**
- Events are immutable after storage
- Threat scores calculated by backend rules
- Alerts generated by backend (not frontend)
- Severity levels assigned by backend

**Frontend is Display Layer Only:**
- Console receives events from WebSocket
- Console displays them in analyst-friendly order
- Console provides drill-down interaction
- Frontend does NOT aggregate, summarize, or calculate threat

---

## Console Behavior

**Real-Time (Instant):**
- New event arrives → appears at top of stream

**On-Demand (Load when analyst clicks):**
- Event details (raw data lookup)
- Related events (query same source IP)
- Historical context (alert history)

**Never (These are gone):**
- Polling for updates (WebSocket is exclusive)
- Metric aggregation (raw events only)
- Summary statistics (no pre-calc)
- Periodic refreshes (stream is continuous)

---

## Key Principle: Event-First

**Don't think:** "System has 200 total events today with 5 alerts"  
**Think:** "Event at 14:32 UTC: Port scan from 10.1.1.100 with threat score 87"

The newest, most relevant event is what matters. Aggregates hide patterns.

---

## Building on This Model

All future UI components will:
1. Display raw events (not summaries)
2. Accept WebSocket input (not REST polling)
3. Support analyst drill-down (details → timeline → context)
4. Show state transitions (OPEN → INVESTIGATING → RESOLVED)
5. Maintain investigation audit trail
